# Rhizome RAG Architecture & Schema

This document outlines the internal workings, database schema, and naming conventions for **Rhizome**, OpenTendril's Retrieval-Augmented Generation (RAG) engine.

## Overview
Rhizome acts as the memory and context-retrieval system for OpenTendril. Rather than vectorizing raw text blocks, Rhizome breaks source code down into structured **Symbols** (functions, structs, classes, dependencies) stored in a local SQLite database.

This "Symbol-Level Local Indexing" ensures that when an LLM asks for context about a specific function, it receives the exact code block, its docstring, and its surrounding dependencies, drastically reducing token usage and hallucination.

## SQLite Schema
Rhizome stores symbols in the `rhizome.db` SQLite database using the following conceptual schema:

- `repositoryName` (TEXT): The namespace of the repo (e.g., `owner/repo`).
- `name` (TEXT): The identifier of the symbol (e.g., `GenerateRepoMap`, `Worker`).
- `type` (TEXT): The structural type of the symbol.
- `filePath` (TEXT): The relative path to the file.
- `lineStart` (INT): Starting line number of the symbol.
- `lineEnd` (INT): Ending line number of the symbol.
- `stubContent` (TEXT, ENCRYPTED): The extracted code block, prepended with its docstring.

## Symbol Naming Convention & Types

Rhizome extractors (`GoParser`, `RegexParser`) classify code blocks into the following `type` strings. **No underscores are used** for standard language constructs.

### Standard Language Symbols
- `function` — Standalone functions (Go, Python, JS/TS).
- `method` — Functions attached to a receiver or class.
- `class` — Object-oriented class definitions.
- `struct` — Data structures (Go).
- `interface` — Contract definitions (Go, TS).
- `type` — Type aliases.

### Pseudo-Symbols
Pseudo-symbols are synthetic constructs generated by Rhizome to provide the LLM with metadata that doesn't correspond to a specific function or class.

- `file_context`
  - **Purpose**: Captures file-level metadata, specifically imported packages and external dependencies.
  - **Name**: Always strictly `"file_context"`.
  - **Generation**: Created exactly once per parsed file.
  - **Sorting**: When `repomap.go` generates a repository map, `file_context` is explicitly sorted to appear **first** beneath the file path header. This guarantees the LLM reads a file's dependencies before analyzing its internal logic.

## Security & Encryption
The `stubContent` (which contains the actual proprietary source code and docstrings) is **symmetrically encrypted** at rest within the SQLite database using AES-GCM (`cmd/stem/internal/rhizome/crypto.go`). This ensures that local context databases do not leak plaintext source code on the filesystem.
